## Зачем нужен Floret

Научить модель вызывать инструмент - лишь начало продукта с агентом. В реальной эксплуатации нужны также надежный цикл модели, долговечная история диалога, выполнение инструментов с подтверждением, работа с длинным контекстом, восстанавливаемые операции и факты runtime, которым может доверять интерфейс.

Floret дает эту runtime-основу, не присваивая себе продукт вокруг нее. Это не UI для агента, не граф workflow и не фреймворк оркестрации множества агентов. Это Go-слой за опытом, который вы создаете.

Это различие важно, когда продукт должен оставаться действительно вашим. Интерфейс, система идентичности, модель прав, маршрутизация моделей, секреты, модель данных и доменные инструменты полностью остаются под вашим контролем. Floret берет на себя сложный жизненный цикл исполнения агента.

### Для продуктов, которым тесно в готовом шаблоне

- **Вы сохраняете контроль над путем к модели.** Используйте встроенную конфигурацию или реализуйте `runtime.ModelGateway`. Floret ведет запросы и их продолжение, а транспорт и учетные данные остаются у продукта.
- **Инструменты меняются вместе с контекстом.** Зарегистрируйте строгие доменные инструменты в `tools.Registry`, затем обновляйте инструменты, хостируемые возможности, инструкции и контекст хоста через `runtime.ToolSurfaceProvider` в безопасных точках запуска.
- **Диалоги становятся надежными активами.** `runtime.Host` управляет потоками, ходами, повторными запусками, ветвлениями, дочерними потоками под управлением родителя и историей, безопасной для provider.
- **Политика подтверждения остается в продукте.** Floret понимает общие эффекты, ресурсы и состояния подтверждения. Кто, что, где и почему может выполнить, решает ваш продукт.
- **Исполнение можно наблюдать.** Подключайте очищенные события, давление контекста, факты сжатия контекста и нейтральные временные шкалы активности к любому UI, не раскрывая промпты, секреты или внутренние записи хранилища.
- **Тестируйте продукт, а не случайность модели.** Fake Provider делает агентные сценарии детерминированными локально и в CI.

## Контроль остается у вас

Граница Floret намеренно узкая: Floret владеет механикой движка, а приложение сохраняет каждое продуктовое решение.

| Floret выполняет | Ваше приложение решает |
| --- | --- |
| Цикл provider, повторы, продолжение инструментов и конечное состояние | Когда пользователи могут начать, повторить, прервать или отменить работу |
| Журнал потока, область промпта, ledger provider и runtime-артефакты | Пользователи, рабочие пространства, биллинг, метаданные и хранение продукта |
| Проверка схемы инструмента, общие эффекты, цикл подтверждения и проекция результата | Авторизация, UI подтверждения, доменные действия и пользовательские тексты |
| Давление контекста, цикл сжатия и история, видимая модели | Какие данные продукта можно передать модели и как их показать |
| Очищенные события и нейтральные факты активности | Макет, сценарии, элементы управления, маршрутизация и политика диагностики |

Поэтому операционная консоль, среда программирования, инструмент поддержки и отраслевой помощник могут пользоваться одной надежной runtime, не превращаясь в один и тот же продукт.

## Быстрый старт

Установите стабильные пакеты для downstream-приложений:

```bash
go get github.com/floegence/floret/config github.com/floegence/floret/runtime github.com/floegence/floret/tools github.com/floegence/floret/observation
```

Запустите долговечный поток с детерминированным Fake Provider:

```go
host, err := runtime.NewHost(runtime.HostOptions{
	Config: config.Config{
		Provider: config.ProviderFake, Model: "fake-model", FakeResponse: "Hello from Floret.",
		AgentProfile: config.AgentProfile{ID: "support-agent", Name: "Support Agent"},
	},
	Store: runtime.NewMemoryStore(),
})
if err != nil { /* handle error */ }
defer host.Close()

thread, err := host.StartThread(ctx, runtime.StartThreadRequest{ThreadID: "thread-1"})
result, err := host.RunTurn(ctx, runtime.RunTurnRequest{
	ThreadID: thread.ID, TurnID: "turn-1", RunID: "run-1",
	Input: "Welcome a new customer in one sentence.",
})
```

Полный выполняемый пример есть в [английском README](README.md#quick-start). Если транспорт модели принадлежит вашему продукту, используйте OpenAI-совместимую конфигурацию или передайте `runtime.ModelGateway`. Чтобы Floret хранил собственные runtime-данные, используйте `runtime.OpenSQLiteStore(path)`; данные продукта остаются в вашем хранилище и связываются через `runtime.ThreadID`.

## Как устроить интеграцию в production

### Дайте runtime только необходимые полномочия

Определяйте доменные действия через `tools.Registry`. У каждого инструмента есть строгая JSON Schema, а также описание эффектов и ресурсов. Floret проверяет вызов, при необходимости запрашивает подтверждение, запускает handler, записывает результат и возвращает его модели. Handler по-прежнему обязан применять авторизацию, характерную для вашего продукта.

Для динамических возможностей пусть `ToolSurfaceProvider` возвращает `runtime.ToolSurface`. Это подходит для режимов продукта, смены рабочего пространства, прав или хостируемых возможностей. Floret обновляет эту поверхность перед запросами к модели и локальной диспетчеризацией, поэтому старое решение модели не сможет незаметно обойти более новую продуктовую политику.

### Явно работайте с идентичностями

- `ThreadID` идентифицирует долговечный диалог.
- `TurnID` идентифицирует один видимый пользователю ход.
- `RunID` идентифицирует конкретное выполнение provider.
- `PromptScopeID` - граница повторного использования кэша промптов и ledger provider.

Эти идентичности намеренно различны. Если позднее требуется завершить ожидающую работу инструмента, принадлежащую хосту, используйте сохраненные Floret `ThreadID`, `TurnID` и `RunID`, а не идентификатор UI, аудита или отображения.

### Отображайте факты, а не внутренности движка

Передайте `runtime.EventSink`, чтобы получать очищенные события жизненного цикла, и используйте DTO из `observation` для давления контекста, сжатия и временных шкал активности. События предназначены для отображения и диагностики в хосте; это не хранилище секретов и не замена базе данных продукта.

Когда нужна долговечная проекция для отображения, используйте `ThreadTurnProjection` и публичные detail API. Не читайте таблицы хранилища Floret напрямую и не собирайте в хосте заново историю, видимую provider.

### Тестируйте уверенно

Fake Provider позволяет проверять поведение инструментов, сценарии подтверждения, повторы, давление контекста и UI-проекции хоста без вызовов реальной модели:

```bash
go test ./...
go run ./cmd/floret-test-ui
```

Локальная тестовая консоль предназначена для участников проекта: в ней можно изучать сессии Fake Provider, очищенные события, сценарии инструментов и хостируемые дочерние потоки. Это не поверхность интеграции для downstream-приложений.

## Справочные материалы

Downstream-приложения должны импортировать только эти публичные пакеты:

```text
github.com/floegence/floret/config
github.com/floegence/floret/runtime
github.com/floegence/floret/tools
github.com/floegence/floret/observation
```

Все, что находится в `internal/`, - деталь реализации. Источником истины для API служит [pkg.go.dev](https://pkg.go.dev/github.com/floegence/floret/runtime); [база знаний OKF](okf/index.md) объясняет участникам архитектуру и словарь runtime.
